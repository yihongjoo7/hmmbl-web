## 1. 코딩 전에 생각하라

** 가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현 전에:
- 가정 사항을 명시적으로 밝혀라. 불확실하다면 물어라.
- 여러 해석이 가능하다면 제시하라. 조용히 선택하지 마라.
- 더 단순한 접근법이 있다면 말하라. 필요할 때는 반론을 제기하라.
- 불명확한 부분이 있다면 멈춰라. 무엇이 헷갈리는지 짚어라. 물어라.


## 2. 단순함을 우선하라

** 문제를 해결하는 최소한의 코드. 추측성 코드는 금지. **

- 요청받은 것 이상의 기능은 추가하지 마라.
- 단일 용도 코드에 추상화를 도입하지 마라.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 넣지 마라.
- 불가능한 시나리오에 대한 오류 처리를 하지 마라.
- 200줄을 썼는데 50줄로 가능하다면, 다시 작성하라.

스스로에게 물어봐라: "시니어 엔지니어가 이걸 보고 지나치게 복잡하다고 할까?" 그렇다면 단순화하라.


## 3. 외과적 변경

** 반드시 필요한 것만 건드려라. 자신이 만든 것만 정리하라. **

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포맷을 "개선"하지 마라.
- 망가지지 않은 것을 리팩토링하지 마라.
- 자신이라면 다르게 했더라도 기존 스타일에 맞춰라.
- 관련 없는 데드 코드를 발견했다면, 언급하라 — 삭제하지 마라.

변경으로 인해 고아 코드가 생겼을 때:
- 자신의 변경으로 인해 쓸모없어진 임포트/변수/함수를 제거하라.
- 요청받지 않은 이상 기존 데드 코드는 제거하지 마라.

기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 한다.


## 4. 목표 중심 실행

** 성공 기준을 정의하라. 검증될 때까지 반복하라. **

작업을 검증 가능한 목표로 전환하라:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성한 후 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 작성한 후 통과시켜라"
- "X 리팩토링" → "전후로 테스트가 통과하는지 확인하라"

다단계 작업에는 간략한 계획을 명시하라:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있어야 독립적으로 반복 실행이 가능하다. 모호한 기준("잘 작동하게 만들어라")은 지속적인 확인을 요구한다.
