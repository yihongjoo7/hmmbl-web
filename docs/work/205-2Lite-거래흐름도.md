# 205. 2-Lite 거래(API) 요청 흐름도 — 정상/오류 시나리오

> 대상: 개발자, QA
> 범위: `NEXT_PUBLIC_DPOP_MODE=native`(2-Lite) 기준. webview 모드는 서명 주체·토큰 발급 경로만 다르므로 갈라지는 지점마다 별도 표기한다.
> 관련: `13-auth-system.md` §9, `14-bridge-guide.md` §9, `15-api-client.md` §1.1, `docs/work/204`
> 소스 기준: `lib/api/apiClient.ts`, `lib/api/fileUploadClient.ts`, `lib/auth/dpop/proofProvider.ts`, `lib/auth/interceptor.ts`, `lib/bridge/bridgeActions.ts`, `app/(protected)/layout.tsx`, `features/auth/hooks/useTokenReceiver.ts`

---

## 1. 참여자 요약

| 참여자 | 파일 | 책임 |
|---|---|---|
| 화면/서비스 | `features/*/services/*Api.ts` | `apiClient.get/post/put/delete` 호출 |
| apiClient | `lib/api/apiClient.ts` | 헤더 빌드, fetch 실행, 401 판정·재시도 |
| proofProvider | `lib/auth/dpop/proofProvider.ts` | DPoP proof 서명 주체 분기(`getDPoPHeader`) |
| interceptor | `lib/auth/interceptor.ts` | 401 시 토큰 갱신 조율(webview/native 분기), pendingQueue |
| bridgeActions | `lib/bridge/bridgeActions.ts` | `window.bridge.*` 호출 + 이벤트 대기 래핑 |
| Bridge | `window.bridge` (네이티브 주입) | `createDpopProof`(동기)·`requestNativeToken`(비동기, 이벤트 응답) |
| Native | 앱(KeyStore + TokenStorage) | DPoP 키 서명, 액세스 토큰 보유·갱신 |
| Server | login-server / 거래 API | DPoP 검증 + 토큰 검증 + 비즈니스 로직 |

두 모드 공통: **fetch는 항상 웹(apiClient)이 직접 수행**한다. 갈라지는 것은 ① DPoP proof를 누가 서명하는지(`getDPoPHeader`), ② 토큰을 누가 발급/갱신하는지(webview=`/auth/token` 코드교환, native=`requestNativeToken`)뿐이다.

## 2. 정상 흐름 (Happy Path)

### 2.1 최초 진입 — 인증 부트스트랩 (`app/(protected)/layout.tsx`)

**native 모드**

```
Protected 화면 mount, isAuthenticated=false
      │
      ▼
resolveDpopMode() === 'native'
      │
      ▼
requestNativeToken()                         ← lib/bridge/bridgeActions.ts
      │  bridgeEventBus.waitFor('tokenReceived', 10s) 등록 후
      ▼
window.bridge.requestNativeToken()
      │
      ▼
Native: TokenStorage 세션 유효 확인(필요 시 자체 refresh)
      │
      ▼
dispatchBridgeEvent('tokenReceived', { access_token, user })
      │
      ├─▶ useTokenReceiver(전역 상시 구독) → setAuth(user, access_token) → isAuthenticated=true
      └─▶ requestNativeToken()의 waitFor → access_token 반환(호출부는 값 미사용, 대기만 종료)
```

**webview 모드 (비교)**

```
Protected 화면 mount, isAuthenticated=false
      │
      ▼
resolveDpopMode() === 'webview'
      │
      ▼
window.bridge.requestAuthCode() → appAuthCode 이벤트({ code })
      │
      ▼
initAuthFromCode(code, setAuth)
      │  createDPoPProof(url,'POST') + POST /auth/token
      ▼
서버 200 { access_token, user } → setAuth(user, access_token) → isAuthenticated=true
```

### 2.2 거래 요청 정상 처리 — 예: `POST /pay/transactions`

```
화면: apiClient.post('/pay/transactions', body)
      │
      ▼
withRefresh(url, 'POST', fn)
      │
      ▼
buildHeaders(`${baseUrl}${url}`, 'POST')
      │  _getToken() → 토큰 있음
      ▼
getDPoPHeader(url, 'POST')                   ← lib/auth/dpop/proofProvider.ts
      │  resolveDpopMode() === 'native'
      ▼
window.bridge.createDpopProof('POST', url)   ← 네이티브 KeyStore 서명(동기, ms 단위)
      │
      ▼
proof(JWT) 문자열 반환
      │
      ▼
fetch(`${baseUrl}${url}`, {
  method: 'POST', credentials: 'include',
  headers: { Authorization: 'DPoP <token>', DPoP: '<proof>', Content-Type: 'application/json' },
  body: JSON.stringify(body),
})
      │
      ▼
Server: DPoP 검증(htm/htu/iat±5분/jti) + 토큰 검증 통과 → 200 { data, message }
      │
      ▼
res.ok=true → res.json() 반환 → 화면
```

webview 모드는 `getDPoPHeader` 내부에서 `createDPoPProof(url, method)`(웹 IndexedDB 키 서명)만 다르고 나머지는 동일하다.

## 3. 오류 시나리오

### 3.1 DPoP proof 서명 실패 (native 모드, 키 없음/네이티브 오류)

```
buildHeaders → getDPoPHeader(url, method)
      │  resolveDpopMode() === 'native'
      ▼
window.bridge.createDpopProof(method, url) → null 반환(키 없음 등)
      │
      ▼
getDPoPHeader: Error('[DPoP] 네이티브 proof 생성 실패(createDpopProof)') throw
      │
      ▼
buildHeaders → withRefresh 내 `await buildHeaders(...)` 지점에서 예외 전파
      │  ⚠️ 이 시점은 fetch 호출 전이므로 401 재시도 로직을 타지 않는다
      ▼
apiClient.post(...) 호출부까지 그대로 throw → 화면/React Query의 onError에서 처리
```

**참고(현재 동작상 한계):** 이 예외는 `ApiError`가 아닌 일반 `Error`이므로, `applyFieldErrors`나 `ErrorCode` 기반 분기를 타지 않는다. 화면단에서 `instanceof ApiError` 체크만 하는 에러 핸들러는 이 케이스를 놓칠 수 있다 — 필요 시 별도 처리 여부를 검토할 것.

### 3.2 액세스 토큰 만료 → 401 → 정상 갱신 → 재시도 성공

```
fetch(...) → 서버가 만료된 토큰으로 401 응답
      │
      ▼
withRefresh: res.status === 401
      │  _onUnauthorized 설정됨
      ▼
_onUnauthorized() 호출                       ← lib/auth/interceptor.ts::onUnauthorized
      │
      ├─ isRefreshing=false → isRefreshing=true
      ▼
resolveDpopMode() === 'native' → refreshViaNative()
      │
      ▼
requestNativeToken() → window.bridge.requestNativeToken()
      │
      ▼
Native: 세션 유효(또는 자체 refresh 성공) → tokenReceived { access_token, user }
      │
      ├─▶ useTokenReceiver → setAuth(user, access_token)  (store 갱신)
      └─▶ requestNativeToken() waitFor → access_token 반환
      │
      ▼
onUnauthorized: drainQueue(newToken) → 대기 중이던 동시 요청들도 함께 해소
      │
      ▼
withRefresh: buildHeaders(...) 재실행(새 토큰 + 새 proof) → fetch 재시도
      │
      ▼
서버 200 → 화면에 정상 반환(화면 입장에서는 지연만 있고 에러를 보지 못함)
```

webview 모드는 `refreshViaNative()` 대신 `refreshViaWebviewCode()`(`requestAuthCode` → `POST /auth/token`)를 탄다. 나머지 재시도 로직은 동일하다.

### 3.3 토큰 갱신 실패 — 네이티브 세션 만료/미인증

```
_onUnauthorized() → refreshViaNative() → requestNativeToken()
      │
      ├─ (A) Native가 { error: 'NOT_AUTHENTICATED' } 페이로드로 tokenReceived 응답
      │        → requestNativeToken(): result.error 존재 → Error('NOT_AUTHENTICATED') throw
      │
      └─ (B) Native가 10초 내 tokenReceived를 보내지 않음(세션 확인 중 행)
               → bridgeEventBus.waitFor 타임아웃 → BridgeTimeoutError throw
      │
      ▼ (A/B 공통)
onUnauthorized의 catch: rejectQueue(err) → pendingQueue 대기자 전원 reject
      │
      ▼
withRefresh의 catch: _onClearAuth?.() 호출(store clearAuth) → err 그대로 재throw
      │
      ▼
화면: apiClient 호출부에서 catch → 401/미인증 처리(예: 로그인 화면 안내)
```

**참고:** 동시에 여러 요청이 401을 받으면, 큐에 있던 요청들도 각자의 `withRefresh` catch 블록에서 `_onClearAuth?.()`를 호출한다 — `clearAuth()`가 여러 번 불릴 수 있으나 idempotent(멱등)하므로 부작용은 없다.

### 3.4 네트워크 단절/타임아웃 (fetch 자체 실패)

```
withRefresh: await fn(headers)   ← fetch(...) 호출
      │
      ▼
오프라인/DNS 실패/타임아웃 → fetch가 TypeError("Failed to fetch") reject
      │
      ▼
withRefresh에는 fetch 호출을 감싸는 try/catch가 없음
      │
      ▼
TypeError가 그대로 apiClient 호출부까지 전파 → ApiError 아님
```

**참고(현재 동작상 한계):** 2-Full 시절에는 `transportError`(OFFLINE/TIMEOUT) → 친절한 `ApiError` 메시지로 매핑됐으나(`docs/work/203`), 2-Lite는 fetch를 직접 쓰므로 이 매핑이 없다. 네트워크 에러 UX가 필요하면 `withRefresh`에 `try/catch` + `fallbackMessages`류 메시지 매핑을 별도로 추가해야 한다(현재 미구현, 이번 작업 범위 밖).

### 3.5 서버 비즈니스 오류 (4xx/5xx)

```
fetch(...) → 서버가 400/403/404/422/500 등으로 응답(res.ok=false, status≠401)
      │
      ▼
withRefresh: if (!res.ok) throw await toApiError(res)
      │  서버 { detail: { code, message, fields? } } 파싱
      ▼
ApiError(status, code, message, fields) throw
      │
      ├─ fields 있음 → 화면이 applyFieldErrors()로 폼 인라인 에러 표시
      └─ fields 없음 → 화면/QueryCache.onError가 토스트 표시(500+는 전역, 4xx는 화면별)
```

### 3.6 동시 다발 401 (여러 요청이 같은 시점에 만료 토큰 사용)

```
요청 A, B, C가 거의 동시에 fetch → 모두 401 수신
      │
      ▼
A의 withRefresh: _onUnauthorized() 호출 → isRefreshing=false→true, 갱신 시작
B, C의 withRefresh: _onUnauthorized() 호출 → isRefreshing=true → pendingQueue에 등록만 하고 대기
      │
      ▼
갱신 성공 → drainQueue(newToken) → B, C의 대기 Promise들도 함께 resolve
      │
      ▼
A, B, C 각자 buildHeaders(새 토큰+새 proof 재생성) → fetch 재시도 → 정상/오류는 각 요청 개별 결과
```

갱신 실패 시에는 `rejectQueue(err)`로 A/B/C 모두 동일 에러로 reject되고(§3.3), 각자의 `withRefresh` catch에서 개별적으로 `_onClearAuth?.()`를 호출한다.

### 3.7 브릿지 능력 없음 — 구버전 앱/일반 브라우저 (native 설정인데 실제로는 webview로 폴백)

```
NEXT_PUBLIC_DPOP_MODE=native 로 빌드됨
      │
      ▼
resolveDpopMode(): isWebView() === false (브라우저)
              또는 typeof window.bridge?.createDpopProof !== 'function' (구버전 앱)
      │
      ▼
'webview'로 폴백
      │
      ▼
getDPoPHeader → createDPoPProof(웹 키) 사용
app/(protected)/layout.tsx 부트스트랩도 동시에 webview 분기(requestAuthCode 흐름)로 폴백
      │
      ▼
이후 모든 거래 요청·401 갱신이 webview 모드 그대로 동작(기능 저하 없이 자동 폴백)
```

### 3.8 파일 업로드 중 401 (`lib/api/fileUploadClient.ts`)

```
uploadFile(file, url, { onProgress })
      │
      ▼
sendUpload(): getDPoPHeader로 헤더 구성 → fetch/XHR multipart 전송
      │
      ▼
응답 401 → _onUnauthorized() 호출(apiClient와 동일 인터셉터 공유, single-flight)
      │
      ▼
성공 → sendUpload() 1회 재시도(새 헤더로 파일 재전송 — 진행률 콜백은 재시도 시 다시 0%부터)
실패 → rejectQueue → 호출부로 에러 전파(§3.3과 동일 경로, onClearAuth 포함)
```

## 5. 테스트 체크리스트 (QA용)

1. native 모드에서 정상 로그인 후 거래 API 1건 호출 시 200 확인, 서버 로그에 네이티브 KeyStore 서명 proof 확인.
2. 액세스 토큰을 인위적으로 만료시킨 뒤 거래 요청 → 자동 갱신 후 재시도 성공 확인(화면 에러 미노출).
3. 네이티브 세션을 로그아웃 상태로 만든 뒤 거래 요청 → 401 처리 후 로그인 화면 유도 확인.
4. 비행기 모드에서 거래 요청 → 앱이 처리되지 않은 예외로 크래시하지 않는지 확인(§3.4, 현재 친절한 메시지 없음이 정상 동작임에 유의).
5. 동일 화면에서 버튼 연타로 동시 요청 3건 이상 발생 후 토큰 만료 유도 → 모두 정상 재시도되는지(중복 로그아웃 없이) 확인.
6. 구버전 네이티브 앱(또는 브라우저)에서 `NEXT_PUBLIC_DPOP_MODE=native` 빌드 실행 → webview 폴백으로 정상 동작하는지 확인.
7. 대용량 파일 업로드 중 토큰 만료 → 재시도 시 진행률이 0%부터 다시 시작하는지 확인(기대 동작).
